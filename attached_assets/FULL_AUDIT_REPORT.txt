SECTION 1: WHAT IS THIS WEBSITE?
================================================================================

TryNex Lifestyle is a custom apparel e-commerce store for Bangladesh.
Customers can browse products (t-shirts, hoodies, mugs, caps), place orders,
track orders, and pay via bKash/Nagad/Rocket/COD.

Admin panel lets you manage products, orders, customers, blog posts, site
settings, and backups.

SECTION 2: THE TECH STACK (What Everything Is Built With)
================================================================================

FRONTEND (What customers see):
  Language:           TypeScript (a safer version of JavaScript)
  Framework:          React 18 (builds the user interface)
  Build Tool:         Vite (compiles and bundles the code for browsers)
  Styling:            Tailwind CSS (utility classes for design)
  UI Components:      shadcn/ui (pre-built buttons, dialogs, forms, etc.)
  Animations:         Framer Motion (smooth transitions and effects)
  Icons:              Lucide (icon library)
  Routing:            Wouter (handles page navigation like /products, /cart)
  Data Fetching:      TanStack React Query (loads data from backend API)
  Forms:              react-hook-form + Zod (form handling + validation)
  SEO:                react-helmet-async (page titles, meta tags)
  State Management:   React Context (cart, wishlist, site settings)

BACKEND (Server that handles data):
  Language:           TypeScript
  Framework:          Express.js (Node.js web server)
  Database:           PostgreSQL (stores all products, orders, settings)
  ORM:                Drizzle (talks to the database using TypeScript)
  Logging:            Pino (server logs)
  File Uploads:       Google Cloud Storage via Replit sidecar (Replit-only)

SHARED CODE (Used by both frontend and backend):
  API Specification:  OpenAPI 3.1 (defines all API endpoints)
  Code Generation:    Orval (auto-generates API hooks and validation)
  Validation:         Zod (validates data on both sides)

PROJECT MANAGEMENT:
  Package Manager:    pnpm (manages all dependencies)
  Structure:          Monorepo (everything in one repository)
  Version Control:    Git + GitHub

SECTION 7: DATABASE (What Data is Stored)
================================================================================

Database: PostgreSQL
Connection: Needs DATABASE_URL environment variable

TABLES:

1. admins
   - id, username, password_hash, created_at
   - Stores admin login credentials
   - Auto-created on first server start

2. products
   - id, name, slug, description, price, discount_price, category_id
   - image_url, images (gallery), sizes, colors, stock
   - featured, rating, review_count, customizable, tags
   - created_at, updated_at
   - Stores all products you sell

3. categories
   - id, name, slug, description, image_url, product_count, created_at
   - Groups products (T-Shirts, Hoodies, Mugs, Caps, Custom Orders)

4. orders
   - id, order_number, customer_name, customer_email, customer_phone
   - shipping_address, shipping_city, shipping_district
   - payment_method, payment_status, status
   - items (JSON array of ordered products), subtotal, shipping_cost, total
   - notes, created_at, updated_at
   - Every customer order is stored here

5. blog_posts
   - id, title, slug, excerpt, content, image_url, author
   - tags, published (true/false), created_at, updated_at
   - Blog articles

6. settings
   - id, key (unique name), value (text), updated_at
   - Key-value store for all site settings
   - Examples: siteName="TryNex Lifestyle", phone="01903426915"

AUTO-SEED: When the database is empty, the server automatically creates:
   - 1 admin account (username: admin)
   - 5 categories
   - 9 sample products (with Unsplash images)
   - Default settings (name, tagline, phone, etc.)

SECTION 9: HOW FILE UPLOADS WORK
================================================================================

Custom design upload flow:
  1. Customer uploads image on product detail page
  2. Frontend calls POST /api/storage/uploads/request-url
  3. Backend generates a "signed URL" using Replit's sidecar service
  4. Frontend uploads file directly to Google Cloud Storage
  5. File URL stored in order items JSON

THE PROBLEM:
  The upload system uses Replit's internal sidecar service at http://127.0.0.1:1106
  This ONLY works when running inside Replit.

  If backend runs on Render, Railway, or anywhere else:
    - Upload URL generation will FAIL
    - Error: "Failed to sign object URL... make sure you're running on Replit"
    - Custom design uploads will NOT work

  Required environment variables (only exist in Replit):
    DEFAULT_OBJECT_STORAGE_BUCKET_ID
    PRIVATE_OBJECT_DIR
    PUBLIC_OBJECT_SEARCH_PATHS

  TO FIX: Need to replace Replit storage with something portable like:
    - Cloudflare R2 (recommended — stays in Cloudflare ecosystem)

SECTION 10: HOW ADMIN LOGIN WORKS
================================================================================

  1. You enter password on /admin/login
  2. Backend hashes it with SHA-256 + salt "trynex_salt_2024"
  3. Compares with ADMIN_PASSWORD env var (default: "Admins@Trynex")
  4. If match → generates a random token (SHA-256 hash)
  5. Token stored in server memory (a JavaScript Map) — expires in 7 days
  6. Token sent back to frontend as JSON + set as httpOnly cookie
  7. Frontend saves token in localStorage
  8. Every admin request sends token in Authorization header
  9. Backend checks if token exists in memory Map

  PROBLEMS:
    - Tokens are in SERVER MEMORY — if server restarts, ALL admin sessions expire
    - On Render free tier, server sleeps after 15 minutes — wakes up = you're logged out
    - Password hashing uses SHA-256 (weak) instead of bcrypt (industry standard)

SECTION 11: HOW DATA FLOWS (Customer Journey)
================================================================================

BROWSING:
  Customer visits / → Home.tsx loads → calls GET /api/products → shows featured products
  Customer clicks product → ProductDetail.tsx → calls GET /api/products/:id → shows details

CART (No server involved):
  Add to cart → saved in browser localStorage → CartContext.tsx manages state
  Cart persists across page refreshes but is device-specific

CHECKOUT:
  1. Customer fills form (name, phone, address, district)
  2. Selects payment: bKash, Nagad, Rocket, or COD
  3. For mobile payment: copies number 01747292277, sends payment, enters last 4 digits
  4. For COD: selects advance payment method + enters last 4 digits
  5. Frontend calls POST /api/orders with all data
  6. Backend creates order in database with status "pending", payment "pending"
  7. Customer gets order number (like "TNX-001234")

ORDER TRACKING:
  Customer enters phone + order number on /track
  Frontend calls POST /api/orders/track
  Backend finds matching order, returns status
  Shows progress: Pending → Processing → Shipped → Delivered

ADMIN ORDER MANAGEMENT:
  Admin views orders at /admin/orders
  Can change order status (pending/processing/shipped/delivered)
  Can change payment status (pending/paid/failed)

SECTION 12: CURRENT DEPLOYMENT SITUATION
================================================================================

WHAT IS DEPLOYED:
  Cloudflare Pages: Frontend only (static HTML/CSS/JS files)
    - Serves the React SPA
    - _redirects makes all URLs load the SPA (/* → /index.html)
    - _headers sets proper caching

WHAT IS NOT DEPLOYED:
  Backend API: NOWHERE — only runs in Replit development
  Database: NOWHERE — only exists in Replit's PostgreSQL
  File Storage: NOWHERE — only works via Replit sidecar

THE RESULT:
  On Cloudflare, when frontend tries to call /api/anything:
    1. There's no API server on Cloudflare
    2. VITE_API_BASE_URL is not set → calls go to same domain
    3. Cloudflare returns the SPA HTML instead of JSON
    4. Frontend gets HTML when expecting JSON → crash/error

  THIS IS WHY THESE THINGS FAIL ON CLOUDFLARE:
    - Product listing (shows nothing — no products from API)
    - Checkout (cannot create orders)
    - Order tracking (cannot find orders)
    - Admin login (cannot authenticate)
    - Admin everything (no API = nothing works)
    - Backup export/import (no API)
    - Blog (no posts load)
    - Settings (no settings load, falls back to hardcoded defaults)
