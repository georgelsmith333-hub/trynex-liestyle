QUESTION 1: What is the exact current production API base URL the live
frontend is calling right now?
================================================================================

ANSWER: NONE — the frontend is calling ITSELF (same domain, relative paths).

Here's exactly what happens:

  File: artifacts/trynex-storefront/src/main.tsx (line 6-7)
    const apiBase = import.meta.env.VITE_API_BASE_URL;
    if (apiBase) setBaseUrl(apiBase);

  File: artifacts/trynex-storefront/src/lib/utils.ts (line 22-25)
    export function getApiUrl(path: string): string {
      const base = import.meta.env.VITE_API_BASE_URL?.replace(/\/+$/, '') ?? '';
      return `${base}${path}`;
    }

  Since VITE_API_BASE_URL is NOT set in Cloudflare Pages:
    - import.meta.env.VITE_API_BASE_URL = undefined
    - setBaseUrl() never runs → base URL stays null
    - getApiUrl('/api/products') returns '/api/products' (relative path)
    - Browser sends request to: https://your-cloudflare-site.pages.dev/api/products
    - Cloudflare has no /api/ backend → returns index.html (SPA fallback)
    - Frontend receives HTML instead of JSON → parse error → crash

  The API calls use two patterns:
    1. Generated hooks (useListProducts, useGetSettings, etc.)
       → go through lib/api-client-react/src/custom-fetch.ts
       → use setBaseUrl() from main.tsx
       → currently: relative paths (no base URL set)

    2. Manual fetch() calls (AdminBackup, AdminCustomers, AdminOrders)
       → use getApiUrl() from lib/utils.ts
       → currently: returns relative paths (base is empty string)

  BOTH patterns result in: relative /api/* calls → Cloudflare serves HTML → failure.

QUESTION 10: For backup/export/import, the exact root cause
================================================================================

  Is it a frontend bug?          NO — Frontend code is correct.
                                 AdminBackup.tsx correctly calls the right endpoints
                                 with proper auth headers and handles responses.

  Is it a backend bug?           NO — Backend code is correct.
                                 routes/backup.ts correctly exports all 5 tables
                                 (orders, products, categories, settings, blogPosts)
                                 and imports with proper conflict handling.

  Is it an auth issue?           NO — Auth wiring is correct.
                                 AdminBackup.tsx sends Bearer token via getAuthHeaders().
                                 Backend requireAdmin middleware checks it properly.

  Is it a file handling bug?     NO — File handling is correct.
                                 Export creates proper JSON, download works via blob URL.
                                 Import reads file, parses JSON, sends as request body.
                                 JSON body limit is set to 50MB in app.ts.

  Is it a deployment/config      YES — THIS IS THE ONLY ROOT CAUSE.
  issue?
                                 The backend API server is not deployed anywhere.
                                 VITE_API_BASE_URL is not set in Cloudflare Pages.

                                 When AdminBackup.tsx calls:
                                   fetch(getApiUrl('/api/admin/backup/export'))
                                 It resolves to:
                                   fetch('/api/admin/backup/export')
                                 Which hits:
                                   https://your-cloudflare-site.pages.dev/api/admin/backup/export
                                 Cloudflare returns:
                                   index.html (the SPA fallback)
                                 Frontend tries to parse HTML as JSON:
                                   CRASH → "Export failed"

                                 Same exact thing happens for CSV export and import.

  VERDICT: 100% deployment/config issue. Zero code bugs.
  FIX: Deploy backend + set VITE_API_BASE_URL. That's it.
