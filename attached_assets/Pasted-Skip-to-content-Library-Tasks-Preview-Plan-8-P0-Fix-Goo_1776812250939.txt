Plan #8 - P0: Fix Google login + Guest account creation on production
#8 - P0 Auth & Checkout Reliability Fix
What & Why
Customers report Google login fails and "Continue as Guest" fails on production (trynexshop.com / Render API). Both flows are critical for checkout conversion — every minute they're broken loses sales. Diagnose, surface real error reasons in the UI, and fix.

Done looks like
A new visitor on production can sign in with Google in one tap and reach /account.
A new visitor on the checkout page can tap "Continue as Guest" and proceed to place an order; their synthetic credentials persist in localStorage.
When auth fails, the UI shows the actual reason (e.g. "Google sign-in is not configured", "Server unreachable", "Email already registered as a regular account") instead of a generic "Failed".
Add a /api/health/auth diagnostic endpoint that returns { google_configured: bool, jwt_secret_present: bool, db_reachable: bool } so future regressions are debuggable in 5 seconds.
Document required Render env vars + DB site_settings rows in replit.md and the deploy checklist.
Out of scope
Replacing Clerk/Replit Auth. Keep the current custom JWT + Google Identity Services + guest flows.
Facebook login (separate flow, currently working).
Email verification flow.
Steps
Diagnose: read all auth code paths end-to-end, list every reason the Google or guest call could 4xx/5xx in production. Confirm against Render logs if accessible.
Surface real errors: every catch path on both client and server must propagate a human-readable message field. Replace generic "Google login failed" with the API's reason or "Google sign-in is temporarily unavailable" when no reason is present.
Add the /api/health/auth diagnostic endpoint (no secrets in the response — booleans only).
Fix any bugs found in the diagnose step (likely candidates: missing GOOGLE_CLIENT_ID in DB settings, missing Render env var, JWT_SECRET fallback, CORS allowlist, guest-sequence race condition, password length validation refusing valid synthetic passwords).
Add 2 happy-path tests (one for /api/auth/google with a stub credential, one for /api/auth/guest ensuring it returns username+password) and run the full build to make sure nothing regressed.
Update replit.md with the env-var checklist (GOOGLE_CLIENT_ID, JWT_SECRET, ADMIN_JWT_SECRET, ALLOWED_ORIGINS) and a one-liner on how to verify auth health post-deploy.

Plan #24 - Fix 3D mockups for every product
#24 - Fix 3D mockups for every product
What & Why
The 3D preview is the headline feature, but right now the customer's design appears on the SIDE of the t-shirt instead of centered on the chest, and other products (hoodie, long sleeve, cap, mug) are similarly misaligned. The fabric also looks plasticky and the touch interaction is not smooth on mobile. Every product must show the design exactly where the user placed it in 2D, with realistic fabric/material and silky-smooth touch controls.

Done looks like
For every product (t-shirt, hoodie, long sleeve, cap, mug), the design rendered in 3D appears in the SAME position, scale, and rotation that the customer placed it in the 2D editor — front shows on the front panel, back shows on the back panel, no wrap to the sides or armpits.
Fabric materials look realistic: subtle weave/cloth normal map, soft matte sheen on cotton products, ceramic gloss on the mug, brim curvature on the cap. Lighting includes a soft key + fill + rim light so the product feels three-dimensional.
Touch controls feel premium: one-finger orbit with momentum, two-finger pinch-to-zoom, double-tap to reset, no jitter, no rubber-banding past the configured limits. Works smoothly on a mid-range phone.
The 3D preview renders identically in: Design Studio, Product Detail page, Cart, Checkout, Order Confirmation, Order Tracking, and Admin order detail.
A clear loading indicator while GLBs/textures load; graceful 2D fallback if WebGL2 is unavailable.
Out of scope
Adding new product types beyond the existing five.
Changing the 2D editor itself (covered by the Design Studio task).
Replacing the GLB models with new sculpts; reuse and re-UV existing meshes if the chest UV is wrong.
Steps
Audit the UV layout of each GLB and the corresponding PrintZone in mockups.tsx so the texture's chest rectangle lands on the chest UV island. If a model's UVs are misaligned, fix them in code (UV transform on the design overlay mesh) so the design lands on the front chest, hoodie front pocket area, sleeve panel, cap front, and mug body window.
Upgrade the design overlay material so the design respects the underlying fabric (subtle multiply blend, slight roughness inheritance) and add cloth/ceramic normal+roughness maps to base materials.
Improve scene lighting (key/fill/rim) and environment map for soft realistic reflections; tune per-product (matte for cotton, glossy for mug).
Tune OrbitControls for smooth touch: enable damping, set damping factor, configure zoom speed for touch, add double-tap-to-reset, ensure no pan, lock vertical angle sensibly per product.
Make every place that renders the 3D preview use the same shared viewer component and material setup so cart, checkout, admin, etc. all look identical.
Add a polished loading skeleton and error/fallback path when WebGL2 is unsupported.

Plan #29 - Full Replit-platform independence
#29 - Full Replit-platform independence
What & Why
Production runs on Render + Cloudflare R2, but the codebase still has Replit-only fallbacks: the object-storage layer falls back to a 127.0.0.1:1106 sidecar when no R2/S3 vars are set, the deployment route hard-codes /home/runner/workspace as the repo root, and a few feature paths (ACL, signed URLs) work only on Replit's GCS-backed sidecar. We need the production deploy to be 100% Replit-free, with R2 as the sole storage backend and zero hidden dependencies on Replit's runtime.

Done looks like
With R2_* env vars set on Render, the server boots, logs [storage] active backend: r2, and every storage operation (upload, download, signed URL, ACL/visibility, move-to-order-folder) works against R2 with no calls to 127.0.0.1:1106.
If R2 vars are missing in production, the server fails fast with a clear error pointing to the exact env vars needed — instead of silently falling back to the sidecar.
The "Push to GitHub" deployment route either reads REPO_ROOT from env (with a sensible default for Render's filesystem) or is feature-flagged off in production.
Signed URLs for customer/admin downloads work on R2 (presigned S3-style URLs).
Visibility model (/public/ vs /uploads/) is documented and the bucket policy expectations are stated.
The health endpoint reports the active storage backend and R2 connectivity.
Out of scope
Migrating existing files from Replit storage to R2 (separate one-time migration if needed).
Switching to a different storage provider (S3 path stays as already-implemented option).
Steps
Make the storage backend selector strict in production: refuse to start if NODE_ENV=production and no R2/S3 backend is configured.
Implement the missing R2 implementations of getObjectEntityFile, trySetObjectEntityAclPolicy, and moveObjectToOrderPrefix (using S3-compatible APIs against R2).
Make the deployment route's REPO_ROOT configurable; default to the Render filesystem path and feature-flag the route off when not in admin/dev.
Extend the health endpoint to surface storage backend status, R2 connectivity, and a list of any sidecar-only code paths still in use.
Document the required Render env vars and the R2 bucket layout (public vs uploads vs orders) in the project readme.

Customers expect uploading an image to "just work" — the background should disappear automatically, every change should be undoable, and on mobile they should be able to scale, rotate, and move the design with their fingers from any side of the layer without fiddly controls. Done looks like

Undo/Redo is visible and reachable on both desktop and mobile, with keyboard shortcuts (Ctrl/Cmd+Z, Shift+Cmd+Z) and a clearly labeled toolbar pair. History covers add/remove/move/scale/rotate/text-edit/background-removal-toggle. A subtle toast confirms what was undone.
Touch on mobile: each selected layer shows large tappable handles on all four corners + four edges (resize) and a top rotate handle, plus the existing pinch/rotate gesture continues to work. Handles are big enough for fingers (≥36px) and don't overlap small layers.
The 2D preview blends the design with the fabric: subtle multiply over fabric texture and a faint inner shadow so flat PNGs no longer look like a sticker pasted on top.
A persistent "Save draft" indicator + autosave continues to work; an explicit "Restore previous design" entry point exists if local draft is found.

3D-side fixes (handled by the 3D mockup task).
Adding new layer types (e.g. shapes) beyond image and text.
Steps
Trigger automatic background removal as soon as an image layer is added (server first, in-browser model fallback). Cache the original pixels on the layer so toggling BG removal off restores them instantly.
Surface Undo/Redo prominently in the mobile toolbar and keep the existing desktop bar; add keyboard shortcuts and a small "Undid: X" toast.
Add edge handles (N/E/S/W) and a clearly visible rotate handle to every selected layer; ensure handles scale up on touch devices and stay within a comfortable hit area.
Add a fabric-blend layer in the 2D compositor (multiply overlay using a fabric noise + faint inner shadow) so designs feel printed, not pasted.

Make the storage backend selector strict in production: refuse to start if NODE_ENV=production and no R2/S3 backend is configured.

Make the deployment route's REPO_ROOT configurable; default to the Render filesystem path and feature-flag the route off when not in admin/dev.
